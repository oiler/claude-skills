# Audit checklist — Phase 5

Run before every package (Phase 5). Every item is pass/fail; fix before packaging.

**Run the validator first.** `uv run ${CLAUDE_SKILL_DIR}/scripts/validate_plugin.py <plugin-dir>` executes every deterministic check below and is the authority for them — fix failures and re-run to exit 0 before touching the rest by hand. Tags on each item say who owns it: **[script]** — the validator decides, don't re-derive by hand; **[script-assisted]** — the validator covers the named sub-checks, the model audits the remainder; **[judgment]** — the model entirely. `--json` emits machine-readable findings keyed to these item numbers.

0. **[judgment] Deviations doc honored.** If the plugin repo carries a recorded-deviations file (convention: `docs/cowork-builder-deviations.md`), read it before anything else. A checklist item that hits a deviation recorded there — dated, reasoned, decided with the plugin owner — passes with a note; don't re-flag it every audit. A new deviation discovered during this audit gets recorded there with its reasoning, not silently accepted.

1. **[script] SKILL.md filename exact.** Every skill directory in the plugin contains a file named exactly `SKILL.md` — not `skill.md`, not `<name>-SKILL.md`. Wrong filename fails silently, no error surfaces.

2. **[script] `plugin.json` valid, at the right path.** The manifest exists at exactly `.claude-plugin/plugin.json` — a `plugin.json` at the plugin root is a fail. `name` is kebab-case, `version` is valid semver (`MAJOR.MINOR.PATCH`), `description` is present and non-empty. If the `claude` CLI is available, run `claude plugin validate <plugin-dir>/.claude-plugin/plugin.json` and require a clean pass. See `distribution.md` §7.

3. **[script-assisted] Descriptions are pushy.** Every skill's `description` field is third person (describes the skill, doesn't instruct it), is trigger-phrase-rich (packs in concrete phrasing a user would actually type), and closes with a `Use when…` clause. A thin or generic description is a fail — undertriggering is the most common skill defect. See `skill-authoring.md`. (Validator covers: description present and non-empty, frontmatter parses. Pushiness, third person, and the Use-when clause are yours.)

4. **[judgment] Progressive disclosure held.** Each `SKILL.md` stays lean — routing/summary content only — with real detail pushed into `references/`. A `SKILL.md` that reads like a full manual instead of a router is a fail.

5. **[judgment] Standalone + supercharged confirmed.** Every command skill has a working zero-connector path: the `## Without connected sources` section covers the zero-connector path explicitly, and that path never hard-fails just because a connector isn't wired. The "with `~~category` connected" path is additive, never a substitute. See `skill-authoring.md` § Standalone + Supercharged.

6. **[script-assisted] `~~category` sync verified.** Every `~~category` placeholder used in a skill body appears, spelled and cased identically, in all of: the skill body itself, `CONNECTORS.md`'s `Placeholder` column (4-column table: Category / Placeholder / Included servers / Other options), the "connectors this plugin uses" section of `README.md`, and a matching server entry in `.mcp.json`. Grep each token across all four locations and confirm no orphans in either direction. No `~~` token appears in any skill's `description` frontmatter — descriptions use plain category language (see `connectors-and-mcp.md`). Every empty-`url: ""` stub in `.mcp.json` is marked with `*` next to its row in `CONNECTORS.md` or `README.md`, with the footnote `` `*` — Placeholder — MCP URL not yet configured `` defined once. See `connectors-and-mcp.md`. (Validator covers: no ~~ token in any description frontmatter. The four-way body/CONNECTORS/README/.mcp.json sync and stub footnotes are yours.)

7. **[script-assisted] Cowork output hygiene held.** For every skill that writes files: output goes to the user's working folder, never the plugin's own install directory or an arbitrary temp path; no relative paths — only a real, resolved path rooted in the working folder; no `open` or `xdg-open` call, and no language implying the file opened itself; the skill's last move on that path is telling the user exactly where the file landed. Skills meant to be schedulable (or likely to run in remote sessions) must also work with **no working folder at all** — outputs surface as session deliverables, the skill says the file's name rather than fabricating a path, and it carries no MANDATORY FIRST STEP folder guard. See `skill-authoring.md` § Cowork output hygiene, `cowork-runtime.md` § Scheduled tasks, and `live-artifacts.md` §1 for the static HTML deliverable case. (Validator covers: no open/xdg-open inside fenced code blocks. Working-folder discipline and the no-folder mode are yours.)

8. **[script-assisted] `${CLAUDE_PLUGIN_ROOT}` used everywhere.** Every intra-plugin path reference — copying `dashboard.html`, pointing at a template, referencing another skill's file — uses `${CLAUDE_PLUGIN_ROOT}`. Grep the plugin tree for hardcoded absolute paths (`/Users/`, `/home/`, `/tmp/`, etc.) or bare relative paths standing in for a plugin-root reference; any hit is a fail. The rule governs runtime file operations (copy, read, execute); markdown cross-links between plugin files — like the CONNECTORS banner's `[CONNECTORS.md](../../CONNECTORS.md)` — are exempt. (Validator covers: hardcoded absolute paths — /Users/, /home/, /tmp/. That runtime file operations actually use ${CLAUDE_PLUGIN_ROOT} is yours.)

9. **[script-assisted] Security checks pass.** No literal secret, API key, or bearer token appears anywhere in `.mcp.json` or skill bodies — every credential is a `${VAR}` environment-variable substitution, and each referenced `${VAR}` is documented in `README.md` (what it is, where to get it). Every remote `.mcp.json` entry (`http` or `sse` type) uses `https://` — no plaintext remotes. Every agent's `tools:` field is either a pinned least-privilege list or deliberately omitted with the mandatory `# tools not restricted — …` comment explaining why; never present-but-empty, never silently omitted. If any skill instructs Claude to create an artifact, the disclosure matches its refresh model: connector-refreshed → user-facing copy says it keeps itself current from their connected tools (Live Artifacts read connectors without approval prompts, so silence quietly widens what the plugin touches); static/authored snapshot → copy says it reflects the latest run and does not update on its own (never "Live"). If the artifact displays untrusted content, the skill states the link contract and the plugin bundles the sanitized-payload boundary (`live-artifacts.md` §2). See `connectors-and-mcp.md` § Security and `agent-playbook.md` §4. (Validator covers: .mcp.json parses. Secrets, https-only remotes, agent tools fields, and artifact disclosure are yours.)

10. **[script] Distribution artifacts match the declared path.** The validator infers the profile from repo shape and enforces its artifact set. Private/individual: minimal `plugin.json` (`name`, `version`, `description`, `author`) — no `CHANGELOG.md` required, concrete product names allowed. **Private plugin that is its own repo — default layout is the self-marketplace:** `.claude-plugin/marketplace.json` beside `plugin.json`, listing the plugin with `source: "./"` (`/plugin marketplace add <repo>` is the only stable local-install path in Claude Code; see `distribution.md` §4). Public: `plugin.json` also carries `license`, `homepage`, `repository`, `keywords`; the plugin lives in (or is) a marketplace repo; `CHANGELOG.md` at the plugin root in keep-a-changelog format; skill bodies genericized to `~~category` with `CONNECTORS.md` present. The model's remaining half: confirm the *intent* matches the shape — a plugin shipping public artifacts while meant to stay private is a fail even when structurally coherent. See `distribution.md` §3–4.

11. **[judgment] Dedup lint clean.** No two skills in the plugin cover overlapping trigger territory, and no stray experimental/scratch skill directory survived into the packaged tree (e.g. a `-v2`, `-draft`, `-old`, or `-copy` suffixed skill left over from iteration). Every shipped skill has a distinct, non-overlapping `description` scope from every other shipped skill.

12. **[judgment] Sensitive-domain disclaimer present where applicable.** If any skill's function touches finance, legal, or health/medical topics, that skill's body carries an explicit disclaimer to the Cowork end user (not investment/legal/medical advice, consult a professional, etc.) in its user-facing output — not buried only in authoring material. Skills outside these domains are exempt; don't add a disclaimer where the subject matter doesn't call for one.

13. **[script] Package deterministically.** Plugin-is-its-own-repo (default): package with the shipped Makefile (`assets/templates/Makefile`, copied in at scaffold time) — `make plugin` builds `dist/<name>-<version>-<sha>.plugin` from git-tracked content via `git archive`; `make verify` asserts the manifest and at least one skill are in the bundle and the denylist leaked nothing. Reproducible, cannot leak untracked files, immune to stale-zip updates by construction. Plugin inside a container marketplace repo: use the canonical zip command, verbatim:

   ```bash
   cd <plugin-dir> && zip -r /tmp/<name>.plugin . -x "*.DS_Store" && cp /tmp/<name>.plugin <outputs>/<name>.plugin
   ```

   `<name>` is `plugin.json`'s `name` field (kebab-case), not the directory name if they've diverged. Zip to `/tmp` first, then copy — never zip directly into an outputs mount, and delete any stale `/tmp/<name>.plugin` first (`zip` updates archives in place). See `distribution.md` §6.

14. **[judgment] State writes are connector-safe.** No skill instructs updating a file or a spreadsheet cell in place through a native connector, and every mutable-state path a skill writes (baselines, caches, memory files) lives in the working folder or uses immutable versioned snapshots in connected storage. Grep skill bodies for state-write instructions and check each against `connectors-and-mcp.md` § Native connectors — capability model; an in-place connector write is a silent-corruption bug, not a style issue.
